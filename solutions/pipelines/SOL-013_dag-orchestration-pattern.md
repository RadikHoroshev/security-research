## Что это
Паттерн оркестрации, заменяющий линейное выполнение (`Process.sequential`) на направленный ациклический граф (DAG). Задачи выполняются параллельно, если у них нет общих зависимостей, а результаты собираются и верифицируются перед финальным отчётом.

## Почему это хорошо
- **Скорость**: Recon и Code Audit выполняются одновременно, а не ждут друг друга.
- **Отказоустойчивость**: Падение одного агента не останавливает весь пайплайн — ветка графа изолируется.
- **Гибкость**: Легко добавлять новые шаги (например, Intel Lookup) без переписывания всей цепочки.
- **Эффективность ресурсов**: Агенты не простаивают в ожидании завершения предыдущих этапов.

## Как работает
1. **Planner**: Анализирует цель и строит JSON-граф зависимостей (узлы = задачи, рёбра = зависимости).
2. **Scheduler**: Запускает все узлы, у которых выполнены входные условия (нет незавершённых зависимостей).
3. **Executor**: Выполняет задачи параллельно через `asyncio.gather()` или пул воркеров.
4. **Verifier**: Проверяет качество выходных данных каждого узла. Если данные не прошли проверку — узел перезапускается с уточнённым промптом.
5. **Reporter**: Собирает валидированные результаты всех веток в единый отчёт.

```text
[Planner]
   ├── [Recon Agent] ──┐
   ├── [Audit Agent] ──┼→ [Event Bus] → [Verifier] → [Reporter]
   └── [Intel Lookup] ─┘
```

## Как применить у нас
1. **Рефакторинг `team.py`**:
   - Заменить `Crew(process=Process.sequential)` на кастомный `DAGRunner`.
   - Определить зависимости задач в `config/agents.yaml` (например, `depends_on: [recon]`).
2. **Внедрение Event Bus**:
   - Создать `knowledge/event_store.json` или таблицу в SQLite для записи статусов узлов.
   - Каждый агент пишет `task_id`, `status`, `evidence_refs` по завершении.
3. **Verifier Agent**:
   - Добавить нового агента, который проверяет наличие артефактов перед передачей данных Reporter'у.
   - Если артефактов нет → retry или fallback на альтернативный инструмент.
4. **Async Execution**:
   - Использовать `asyncio` для параллельного запуска независимых задач.
   - Добавить `timeout` и `retry` политику для каждого узла.

## Зависимости
- Python 3.11+ (`asyncio`)
- Рефакторинг `security-team/team.py`
- Обновление `config/agents.yaml` (добавление поля `depends_on`)
- `networkx` (опционально, для визуализации графа)

## Ограничения
- **Сложность отладки**: Параллельное выполнение труднее логировать и воспроизводить.
- **Конфликты ресурсов**: Несколько агентов могут одновременно обращаться к одному инструменту (нужен rate-limit).
- **Требует дисциплины**: Каждая задача должна чётко описывать входные и выходные данные, иначе граф не соберётся.
